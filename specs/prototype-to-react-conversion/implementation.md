## How to read this plan

Each step below is a logically coherent, independently reviewable unit. Steps build forward — no step depends on a later step. Each step lists:

- **Summary**: what and why
- **Files affected**: paths and what changes
- **Estimated diff size**: ~lines, soft target ~500 LOC per step (CSS counts)
- **Details**: code snippets where they pin a non-obvious decision; pointers to the prototype HTML where the work is "port the same markup/CSS, in React/CSS-Modules form."

For pure ports (markup + CSS verbatim from the prototype), the spec does **not** repeat the source — the prototype HTML at `/home/doug/docs/datagoat-2026-04-27.html` is the source of truth. The spec pins down anything that requires judgment: decomposition, prop shapes, integration with contexts, accessibility wiring, and changes from the prototype.

The prototype's vanilla-JS data structures (`window._wellnessData`, `_profileVariants`, etc.) are extracted into typed modules under `src/data/` and `src/metrics/` — see the "Foundations: typed data + metric definitions" step.

## CSS porting strategy

Every component step in this plan ports the corresponding prototype CSS rules into the component's `.module.css` file. The pattern, applied uniformly:

- **Markup is ported by element ID or class** (e.g., "port `#login-panel`"). The prototype HTML is searchable by these identifiers, so no line numbers are quoted — they would drift as the prototype is updated.
- **CSS rules are ported by class name**. Each step lists the prototype class names that move into its CSS Module under "CSS classes ported." The prototype uses BEM-ish kebab-case (`.cta-btn`, `.field-input`, `.section-cal`); when ported into a `.module.css` file, the class name itself is rewritten to **camelCase in the CSS source** (`.ctaBtn`, `.fieldInput`, `.sectionCal`) so JSX consumers can reference them as `css.ctaBtn` without bracket-access — this is the convention this codebase uses across every CSS Module (including the pre-existing `common.module.css`). Vite's CSS Modules default doesn't auto-convert kebab to camel, and bracket access (`css['cta-btn']`) is uglier than camelCase at every call site, so we make the conversion in the CSS file. Reviewers verifying a port grep the kebab-case prototype class name, then look for the camelCased equivalent in the matching `.module.css`.
- **Hardcoded values that match a token are replaced** with `var(--token)` references. (e.g., the prototype's `transition: background 150ms ease` becomes `transition: background var(--dur-quick) var(--ease-default)`.)
- **Decorative / ambient `@keyframes` and long transitions are wrapped in `@media (prefers-reduced-motion: reduce)` blocks** that set them to `none` — per requirements "Reduced motion."
- **Shared utility classes** (`.cta-btn`, `.cta-btn-secondary`, `.field-wrap`, `.field-input`, `.field-label`, `.field-error-msg`, `.field-hint`, `.field-select`, `.field-info-btn`, `.skip-link`, `.section-heading`, `.nav-menu-btn`, `.nav-home-btn`, `.back-nav-btn`, `.eye-btn`) live in shared CSS Modules introduced in the steps that first need them; subsequent steps import the shared module rather than re-porting the rule. Ownership is called out per step.

A reviewer can verify the port by grepping each prototype class name and checking it appears (camelCased) in exactly one `.module.css` file.

*Icon extraction process.* The prototype has 113 `<svg>` instances; ~30 are unique glyphs (the same Edit pencil shows up in many places, etc.). Pin the extraction in a small Node script `tools/extract-icons.mjs` that ships in this same step:

```js
// tools/extract-icons.mjs
import { readFileSync, writeFileSync, mkdirSync } from 'node:fs';
import { createHash } from 'node:crypto';

const html = readFileSync('/home/doug/docs/datagoat-2026-04-27.html', 'utf8');
const svgs = [...html.matchAll(/<svg[^>]*>[\s\S]*?<\/svg>/g)].map(m => m[0]);
const unique = new Map(); // hash -> svg
for (const svg of svgs) {
  const inner = svg.replace(/^<svg[^>]*>|<\/svg>$/g, '').trim();
  const hash = createHash('sha1').update(inner).digest('hex').slice(0, 8);
  if (!unique.has(hash)) unique.set(hash, svg);
}
mkdirSync('src/icons', { recursive: true });
for (const [hash, svg] of unique) {
  // Strip wrapper width/height; svgr forwards props at consumer site
  const cleaned = svg.replace(/\s+(width|height)="[^"]*"/g, '');
  writeFileSync(`src/icons/icon-${hash}.svg`, cleaned);
}
console.log(`Extracted ${unique.size} unique glyphs from ${svgs.length} instances.`);
```

Run with `node tools/extract-icons.mjs`. Output is `src/icons/icon-{hash}.svg` files; rename each to its canonical kebab-case role name (`home.svg`, `hamburger.svg`, etc. — the implementor inspects each glyph and picks the name). The dotall `[\s\S]*?` pattern works without regex flags and matches across newlines, which is what `grep -oE` couldn't do. **Don't use bash `grep -oE`** — it operates line-by-line and silently skips multi-line SVGs (most of them in this prototype), so an implementor running the obvious-looking one-liner would extract a fraction of the icons and conclude the prototype has fewer than it does.

*Brand SVG extraction.* The three brand marks are already URL-imported in the prototype (`<img src="icons/google-logo.svg">` etc.). Pull the corresponding files from the designer's `icons/` folder (delivered alongside the HTML) and drop them into `public/icons/`. If the designer hasn't delivered the source files separately, extract them from the prototype's inline `<svg>` (Login screen only — they're not inlined elsewhere).

*Font sourcing.* Use Google Fonts' `&subset=latin` URL or `glyphhanger` to generate the Latin-subset WOFF2 files. The four faces are: Barlow Regular 400, Barlow Bold 700, Barlow Condensed SemiBold 600, Barlow Condensed ExtraBold 800. License: Barlow ships under the SIL Open Font License — copy `LICENSE-Barlow.txt` into `public/fonts/` per OFL terms.

*`@font-face` block:*


### Migration framework + tests

**Summary**: Stand up the Firestore-document migration layer described in requirements "Data Versioning and Migration." Migrations are pure functions registered by `(docType, fromVersion)`; `migrateDocument()` walks the chain to current. This step also lands Vitest config (used by the previous step's tests too — but Vitest is a zero-config layer over Vite, so it Just Works once `package.json` has the script). No registered migrations exist yet (this is v1 for every doc type), but the framework + fixtures + tests ship now so the seam is exercised before any future migration lands.

**Files affected**:
- `vitest.config.ts` — `defineConfig({ test: { environment: 'node', globals: false, setupFiles: ['./src/test/setup.ts'] } })`. Vitest picks up `*.test.ts` and `*.test.tsx` anywhere under `src/` by default. Pure-logic tests (migrations, dates, completeness, Cloud Functions) run in node for fast startup; component tests opt into jsdom via a `// @vitest-environment jsdom` comment at the top of each test file (Vitest's per-file environment override). Keeps the default fast and explicit per-file rather than slowing every test with jsdom boot.
- `src/test/setup.ts` — `import '@testing-library/jest-dom'` (extends `expect` with DOM matchers like `toBeInTheDocument`, `toHaveAttribute`, `toBeVisible`); plus a small set of jsdom polyfills component tests need: a `window.matchMedia` mock that exposes `addEventListener` / `removeEventListener` (jsdom's stub doesn't, and the DashboardHeaderSlide carousel + reduced-motion guards both use it), and an `IntersectionObserver` no-op stub if any component test ends up needing one (lazy-add when first surfaced — don't pre-add); and a global `afterEach(() => { localStorage.clear(); sessionStorage.clear(); })` so VerificationBanner's per-uid dismiss test (and any future storage-touching test) can assert cross-test isolation without per-file boilerplate. Lands here so component tests in later steps just work.
- `src/test/router.tsx` — exports a `renderWithRouter(ui, { initialEntries?: string[] })` helper that wraps RTL's `render` with `<MemoryRouter initialEntries={initialEntries ?? ['/']}>` + `<Routes><Route path="*" element={ui} /></Routes>`. All component tests that need routing context (`LoginForm`, `SignupForm`, `WellnessLog`, `ProtectedRoute`, etc.) consume this rather than re-deriving the wrapper boilerplate. Pins one router-mock pattern across the test suite. *Convention*: tests that need to assert on `useNavigate` calls consume `renderWithRouter` plus a separate `vi.mock('react-router-dom', () => ...)` to capture navigate calls — don't double-mount or wrap in a real `<BrowserRouter>`.

*Fake-timer convention.* Tests using `vi.useFakeTimers()` (`WellnessLog.test.tsx`, `DashboardHeaderSlide.test.tsx`) must wrap timer advancement in `await act(async () => { vi.advanceTimersByTime(N) })` to avoid React 18/19 act warnings — bare `vi.advanceTimersByTime` schedules state updates outside React's batching scope. The convention is pinned here so test authors don't rediscover it per file.

*CSS-rule assertions.* jsdom does not compute styles from CSS Modules — assertions like `toHaveStyle('pointer-events: none')` are unreliable. *Why `beforeUserCreated` and not `beforeSignedIn`.* `beforeUserCreated` fires on the first auth attempt for a given identity. On their next sign-in, `beforeSignedIn` could re-validate. **Decision**: don't implement `beforeSignedIn` for this; if it becomes an issue, follow-up PR.

*Trigger code shape:*

```ts
// functions/src/auth/blockFacebookMissingEmail.ts
import { beforeUserCreated, HttpsError } from 'firebase-functions/v2/identity';
import { defineString } from 'firebase-functions/params';

// Kill switch — set FACEBOOK_BLOCKER_ENABLED=false in the Firebase console
// (Functions → Configuration) to disable this trigger without a code redeploy.
// Reads at cold-start; existing instances pick up the change on next start.
const BLOCKER_ENABLED = defineString('FACEBOOK_BLOCKER_ENABLED', { default: 'true' });

// The [BLOCKED_NO_EMAIL] sentinel is the client-side discriminator for this
// rejection; the rest of the message is the user-facing copy. Keep the sentinel
// stable even if the copy is edited — the client matches on the sentinel only.
export const blockFacebookMissingEmail = beforeUserCreated((event) => {
  if (BLOCKER_ENABLED.value() !== 'true') return; // kill-switch
  const providers = event.data?.providerData ?? [];
  const isFacebook = providers.some((p) => p.providerId === 'facebook.com');
  if (isFacebook && !event.data?.email) {
    throw new HttpsError(
      'invalid-argument',
      '[BLOCKED_NO_EMAIL] Your Facebook account does not share an email address with us. Either share your email with Facebook, or sign up with a different method.',
    );
  }
});
```

*Kill switch — when and how to flip it.* The `FACEBOOK_BLOCKER_ENABLED` parameter exists for one scenario: an unexpected sign-up regression that traces to this trigger (a false-positive rejection of legitimate Facebook sign-ins, a throw on a non-Facebook provider, a runtime error inside the function). Because the trigger sits on the auth critical path, a malfunctioning blocker breaks new-user sign-up for every provider until disabled. To flip: set `FACEBOOK_BLOCKER_ENABLED=false` via the Firebase console (Functions → Configuration → edit parameter) or via CLI (`firebase functions:config:set` is the v1 idiom; v2 params are configured via the console or `firebase deploy --only functions` after editing the default in code). Existing function instances pick up the new value on next cold-start; an explicit redeploy speeds it up. To re-enable, set back to `'true'`. The trigger's unit tests cover the kill-switch path so a future refactor can't accidentally remove it.

*Sentinel rationale.* The Firebase Auth client SDK surfaces blocking-function rejections as `auth/internal-error` with the trigger's message string concatenated into `error.message` — there is no structured error code. Matching on a stable sentinel (`[BLOCKED_NO_EMAIL]`) instead of substring-matching the user-facing copy makes the client robust to two failure modes: (a) future SDK changes to how blocking errors are wrapped (the v8→v9 migration changed the wrapping shape once already), and (b) editing the user-facing copy in this function without remembering to update the client. The user-facing text comes from stripping the sentinel and rendering the remainder, so the function is the single source of truth for the copy.

*Client-side error mapping.* When the trigger throws, the client sees an error with `code: 'auth/internal-error'` and `message` containing the sentinel-prefixed text. The auth-screens step's `signInWithProvider` helper checks for the sentinel substring and returns `{ ok: false, kind: 'blocked-no-email', message: <stripped text> }`; the LoginForm/SignupForm OAuth handler renders `result.message` directly. No entry in `authErrorMessages` for this case.

*Emulator coverage.* `firebase emulators:start --only auth,firestore,functions` runs the blocking function in the Auth emulator. The repo's existing `npm run emulators` command should be extended to include functions so local dev exercises the trigger.

**Details**:

*Why per-date docs vs one big array.* Firestore docs have a 1MB limit and reads are cheaper for small docs. Per-date docs scale gracefully to multi-year datasets. The collection `users/{uid}/wellnessEntries` is fetched in full by DataContext (it's small — 30 docs at HISTORY=29 in the prototype, will grow but stays small for years).

*Date param normalization.* `useSearchParams()` returns strings; parse with `new Date(YYYY-MM-DD)` and convert to offset via `dateOffsetFromISO(iso)`. If parse fails or offset is out of `[0, HISTORY]`, `<Navigate to="/wellness" replace />` to fall back to today (without the param).

*Debounced writes (DataContext-level).* Both `setWellnessEntry` and `setPerformanceEntry` live on `DataContext`. Each call:

1. **Synchronously merges the partial into a `pendingMap` state** (one map per collection, keyed by date) — a `useState`, not a ref, so every `useData()` consumer re-renders.
2. **Schedules a debounced Firestore write** (500ms) per date. The timer map is keyed by date, so typing on date A then date B within 500ms produces two independent flushes — not one collapsed write that mixes A's and B's partials.
3. The exposed `wellness` / `performance` getters merge server entries (from `onSnapshot`) over `pendingMap`. Consumers see optimistic state synchronously.
4. When `onSnapshot` returns, **field-level reconciliation** drops any pending fields the server has now confirmed. Object-valued fields (`availability`, `performance.metrics`) reconcile via deep equality — `onSnapshot` deserializes fresh references each emission, so reference-identity would never reconcile and pending would grow without bound. `performance.metrics` reconciles per metric key.
5. The Firestore write is fire-and-forget with a `.catch` that calls `logError`. Pending state is **not** dropped at flush time — reconciliation is the sole authority. This preserves the optimistic value across transient `setDoc` failures (network blip, brief offline); the next keystroke flushes a fresh write. On permanent failures (quota, persistent rules denial), the optimistic value stays visible indefinitely; user-facing failure surfacing (toast, retry-with-backoff) is deliberate follow-up.

   **Offline durability**: `firebase.ts` initializes Firestore with `persistentLocalCache({ tabManager: persistentMultipleTabManager() })`, so `setDoc` calls made while offline queue durably to IndexedDB and replay on reconnect — even after tab close or page reload. Without persistent cache the queue lives in memory and is lost on tab close, which is the wrong tradeoff for a PWA where users log data sideline-without-signal. The multi-tab manager is required so a user with both the installed PWA and a browser tab open doesn't have one of them silently degrade to memory-only. The mutation queue is FIFO per document, and our `{ merge: true }` writes use field-level last-writer-wins, so multiple offline edits to the same date coalesce correctly. **Caveat (out of scope)**: when offline with persistent cache, `onSnapshot` fires from the cache with `metadata.hasPendingWrites === true`; reconciliation currently doesn't filter on that flag, so pending entries clear as soon as the local cache reflects them. If a queued write later rejects on server replay, the optimistic UI will diverge from server state. Real exposure is rare (rules denial mid-session is the only realistic trigger); a future fix would be `if (snap.metadata.hasPendingWrites) return;` in the reconciliation effect.
6. Each pending entry stores the queued **uid** alongside the partial. Both flush paths (timer-fired and provider-unmount cleanup) read uid back from the pending entry rather than from the latest `user` reference. After sign-out, a late-firing flush attempt uses the prior uid; Firestore rules reject it (auth context no longer matches the path), so the write is harmless. This is defense-in-depth — the [user]-effect cleanup also clears pending and timers on user change.
7. On provider unmount, a dedicated cleanup effect (declared first so its cleanup runs first under React's declaration-order cleanup) flushes all pending dates synchronously. The `[user]`-effect cleanups that follow find empty refs and become no-ops. On user change (sign-out or user switch), the [user]-cleanup runs alone — pending is cleared, not flushed.

The log components (`WellnessLog`, `PerformanceLog`) drop their per-instance accumulators entirely. They become thin: they read `currentEntry` from `useData()` (which now includes optimistic overlays) and call `setWellnessEntry(date, partial)` per keystroke. No local debounce machinery, no `liveValue` / `liveMetric` overlays, no date-change or unmount flush effects.

*Single-source empty-entry factories.* `emptyWellnessEntry(date)` and `emptyPerformanceEntry(date)` live in `src/types/data.ts`, stamped with `CURRENT_WELLNESS_ENTRY_VERSION` / `CURRENT_PERFORMANCE_ENTRY_VERSION`. Both DataContext (for the optimistic merge — `byDate.set(date, { ...base, ...partial })` requires `base` to be a complete entry when no server doc exists yet) and the log components (for "no entry yet for this date" rendering) import the same factory. The synthesized empty entry never reaches Firestore (only the partial passed to `setWellnessEntry` is written, with version stamped at the firestore-helper boundary), but pinning the current version at the factory removes a footgun for any future consumer that branches on `entry.version`.

*Performance merge memo deep-merge.* The optimistic memo for performance must explicitly deep-merge `metrics`. A symmetric `{...base, ...partial}` (the shape used for wellness) would clobber `base.metrics` with the sparse `partial.metrics` — established metrics for adjacent fields the user is not currently typing on would vanish from totals/dashboard mid-typing and reappear on flush. The memo writes `metrics: { ...(base.metrics ?? {}), ...(partial.metrics ?? {}) }`. Verbatim port of the deleted `mergePartials` invariant.

*Why optimistic state.* The 500ms debounce remains as a Firestore write-amp optimization, but it is no longer on the UI's critical path. Typing in WellnessLog / PerformanceLog updates the chip, the Performance Total column, the dashboard ActivityCalendar cells, the dashboard CTA copy, and any other consumer of `useData()` synchronously per keystroke. Write-amp savings are preserved; the UI-critical-path coupling is decoupled.

*`status: "loaded"` semantics.* With the optimistic memo, `wellness.status === "loaded"` means "we have at least one entry to show — server, pending, or both." It flips to `loaded` the moment any pending entry exists, even before the initial server snapshot returns. Today's enumerated consumers (chip, totals, calendar, charts, CodapPlugin) all want the optimistic semantics. A future spinner-adding consumer must NOT use this status as a proxy for "history fully loaded" — read the underlying server stream via a separate seam if the distinction matters.

*Chip state recomputation.* The chip rerenders synchronously per keystroke because `WellnessLog` reads `currentEntry` from `useData()` and that value reflects the optimistic overlay immediately. No debounce on the chip's critical path.

*MetricInputRow local string state.* The `NumericInput` inside `MetricInputRow` holds the user's exact keystrokes in local state. The parent's controlled `value` round-trips through `Number(raw) → String(numeric)`, which strips trailing decimal points (`"1."`), leading zeros (`"07"`), and bare `"0"` (the `> 0` filter blanks it). Local state preserves what the user typed; the local-vs-parent reconciliation effect accepts external prop changes only when they don't round-trip to the user's current local string (cross-tab edit, form reset).

The script uses a dotall pattern (`/<svg[^>]*>[\s\S]*?<\/svg>/g`) to match across newlines, deduplicates unique glyphs by SHA-1 of inner content, strips wrapper `width`/`height` attrs (svgr forwards props at consumer site), and writes one `icon-{hash}.svg` file per unique glyph to `src/icons/` for the implementor to rename to canonical kebab-case names. Explicit "don't use `grep -oE`" warning preserves the trap so a future maintainer doesn't reinvent the broken approach. Script ships in the same Foundations step alongside the icons it produces.
