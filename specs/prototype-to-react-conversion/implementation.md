## How to read this plan

Each step below is a logically coherent, independently reviewable unit. Steps build forward — no step depends on a later step. Each step lists:

- **Summary**: what and why
- **Files affected**: paths and what changes
- **Estimated diff size**: ~lines, soft target ~500 LOC per step (CSS counts)
- **Details**: code snippets where they pin a non-obvious decision; pointers to the prototype HTML where the work is "port the same markup/CSS, in React/CSS-Modules form."

For pure ports (markup + CSS verbatim from the prototype), the spec does **not** repeat the source — the prototype HTML at `/home/doug/docs/datagoat-2026-04-27.html` is the source of truth. The spec pins down anything that requires judgment: decomposition, prop shapes, integration with contexts, accessibility wiring, and changes from the prototype.

The prototype's vanilla-JS data structures (`window._wellnessData`, `_profileVariants`, etc.) are extracted into typed modules under `src/data/` and `src/metrics/` — see the "Foundations: typed data + metric definitions" step.

## CSS porting strategy

Every component step in this plan ports the corresponding prototype CSS rules into the component's `.module.css` file. The pattern, applied uniformly:

- **Markup is ported by element ID or class** (e.g., "port `#login-panel`"). The prototype HTML is searchable by these identifiers, so no line numbers are quoted — they would drift as the prototype is updated.
- **CSS rules are ported by class name**. Each step lists the prototype class names that move into its CSS Module under "CSS classes ported." The prototype uses BEM-ish kebab-case (`.cta-btn`, `.field-input`, `.section-cal`); when ported into a `.module.css` file, the class name itself is rewritten to **camelCase in the CSS source** (`.ctaBtn`, `.fieldInput`, `.sectionCal`) so JSX consumers can reference them as `css.ctaBtn` without bracket-access — this is the convention this codebase uses across every CSS Module (including the pre-existing `common.module.css`). Vite's CSS Modules default doesn't auto-convert kebab to camel, and bracket access (`css['cta-btn']`) is uglier than camelCase at every call site, so we make the conversion in the CSS file. Reviewers verifying a port grep the kebab-case prototype class name, then look for the camelCased equivalent in the matching `.module.css`.
- **Hardcoded values that match a token are replaced** with `var(--token)` references. (e.g., the prototype's `transition: background 150ms ease` becomes `transition: background var(--dur-quick) var(--ease-default)`.)
- **Decorative / ambient `@keyframes` and long transitions are wrapped in `@media (prefers-reduced-motion: reduce)` blocks** that set them to `none` — per requirements "Reduced motion."
- **Shared utility classes** (`.cta-btn`, `.cta-btn-secondary`, `.field-wrap`, `.field-input`, `.field-label`, `.field-error-msg`, `.field-hint`, `.field-select`, `.field-info-btn`, `.skip-link`, `.section-heading`, `.nav-menu-btn`, `.nav-home-btn`, `.back-nav-btn`, `.eye-btn`) live in shared CSS Modules introduced in the steps that first need them; subsequent steps import the shared module rather than re-porting the rule. Ownership is called out per step.

A reviewer can verify the port by grepping each prototype class name and checking it appears (camelCased) in exactly one `.module.css` file.

**Details**:

*Icon extraction process.* The prototype has 113 `<svg>` instances; ~30 are unique glyphs (the same Edit pencil shows up in many places, etc.). Pin the extraction in a small Node script `tools/extract-icons.mjs` that ships in this same step:

```js
// tools/extract-icons.mjs
import { readFileSync, writeFileSync, mkdirSync } from 'node:fs';
import { createHash } from 'node:crypto';

const html = readFileSync('/home/doug/docs/datagoat-2026-04-27.html', 'utf8');
const svgs = [...html.matchAll(/<svg[^>]*>[\s\S]*?<\/svg>/g)].map(m => m[0]);
const unique = new Map(); // hash -> svg
for (const svg of svgs) {
  const inner = svg.replace(/^<svg[^>]*>|<\/svg>$/g, '').trim();
  const hash = createHash('sha1').update(inner).digest('hex').slice(0, 8);
  if (!unique.has(hash)) unique.set(hash, svg);
}
mkdirSync('src/icons', { recursive: true });
for (const [hash, svg] of unique) {
  // Strip wrapper width/height; svgr forwards props at consumer site
  const cleaned = svg.replace(/\s+(width|height)="[^"]*"/g, '');
  writeFileSync(`src/icons/icon-${hash}.svg`, cleaned);
}
console.log(`Extracted ${unique.size} unique glyphs from ${svgs.length} instances.`);
```

Run with `node tools/extract-icons.mjs`. Output is `src/icons/icon-{hash}.svg` files; rename each to its canonical kebab-case role name (`home.svg`, `hamburger.svg`, etc. — the implementor inspects each glyph and picks the name). The dotall `[\s\S]*?` pattern works without regex flags and matches across newlines, which is what `grep -oE` couldn't do. **Don't use bash `grep -oE`** — it operates line-by-line and silently skips multi-line SVGs (most of them in this prototype), so an implementor running the obvious-looking one-liner would extract a fraction of the icons and conclude the prototype has fewer than it does.

*Brand SVG extraction.* The three brand marks are already URL-imported in the prototype (`<img src="icons/google-logo.svg">` etc.). Pull the corresponding files from the designer's `icons/` folder (delivered alongside the HTML) and drop them into `public/icons/`. If the designer hasn't delivered the source files separately, extract them from the prototype's inline `<svg>` (Login screen only — they're not inlined elsewhere).

*Font sourcing.* Use Google Fonts' `&subset=latin` URL or `glyphhanger` to generate the Latin-subset WOFF2 files. The four faces are: Barlow Regular 400, Barlow Bold 700, Barlow Condensed SemiBold 600, Barlow Condensed ExtraBold 800. License: Barlow ships under the SIL Open Font License — copy `LICENSE-Barlow.txt` into `public/fonts/` per OFL terms.

*`@font-face` block:*

```css

  font-family: 'Barlow';

  font-display: swap;
  src: url('/fonts/barlow-regular.woff2') format('woff2');

/* ...same pattern for the other three faces... */
```

Pure-logic tests (migrations, dates, completeness, Cloud Functions) run in node for fast startup; component tests opt into jsdom via a `// @vitest-environment jsdom` comment at the top of each test file (Vitest's per-file environment override). All component tests that need routing context (`LoginForm`, `SignupForm`, `WellnessLog`, `ProtectedRoute`, etc.) consume this rather than re-deriving the wrapper boilerplate. Component tests assert attribute-level state (`aria-*`, `disabled`, `tabindex`) and class-name presence (`toHaveClass(css.foo)`); CSS rule fidelity is verified manually in the viewport-matrix QA step (Step 18). Pinned here so the pattern doesn't recur as a per-file judgment call.

*Why `beforeUserCreated` and not `beforeSignedIn`.* `beforeUserCreated` fires on the first auth attempt for a given identity. If the function rejects, no user record is created. `beforeSignedIn` would fire on every sign-in and is overkill — once the user is rejected at creation, they never have a record to sign back in with. *Trigger code shape:*

```ts
// functions/src/auth/blockFacebookMissingEmail.ts
import { beforeUserCreated, HttpsError } from 'firebase-functions/v2/identity';
import { defineString } from 'firebase-functions/params';

// Kill switch — set FACEBOOK_BLOCKER_ENABLED=false in the Firebase console
// (Functions → Configuration) to disable this trigger without a code redeploy.
// Reads at cold-start; existing instances pick up the change on next start.
const BLOCKER_ENABLED = defineString('FACEBOOK_BLOCKER_ENABLED', { default: 'true' });

// The [BLOCKED_NO_EMAIL] sentinel is the client-side discriminator for this
// rejection; the rest of the message is the user-facing copy. Keep the sentinel
// stable even if the copy is edited — the client matches on the sentinel only.
export const blockFacebookMissingEmail = beforeUserCreated((event) => {
  if (BLOCKER_ENABLED.value() !== 'true') return; // kill-switch
  const providers = event.data?.providerData ?? [];
  const isFacebook = providers.some((p) => p.providerId === 'facebook.com');
  if (isFacebook && !event.data?.email) {
    throw new HttpsError(
      'invalid-argument',
      '[BLOCKED_NO_EMAIL] Your Facebook account does not share an email address with us. Either share your email with Facebook, or sign up with a different method.',
    );
  }
});
```

*Kill switch — when and how to flip it.* The `FACEBOOK_BLOCKER_ENABLED` parameter exists for one scenario: an unexpected sign-up regression that traces to this trigger (a false-positive rejection of legitimate Facebook sign-ins, a throw on a non-Facebook provider, a runtime error inside the function). Because the trigger sits on the auth critical path, a malfunctioning blocker breaks new-user sign-up for every provider until disabled. To flip: set `FACEBOOK_BLOCKER_ENABLED=false` via the Firebase console (Functions → Configuration → edit parameter) or via CLI (`firebase functions:config:set` is the v1 idiom; v2 params are configured via the console or `firebase deploy --only functions` after editing the default in code). Existing function instances pick up the new value on next cold-start; an explicit redeploy speeds it up. To re-enable, set back to `'true'`. The trigger's unit tests cover the kill-switch path so a future refactor can't accidentally remove it.

*Sentinel rationale.* The Firebase Auth client SDK surfaces blocking-function rejections as `auth/internal-error` with the trigger's message string concatenated into `error.message` — there is no structured error code. Matching on a stable sentinel (`[BLOCKED_NO_EMAIL]`) instead of substring-matching the user-facing copy makes the client robust to two failure modes: (a) future SDK changes to how blocking errors are wrapped (the v8→v9 migration changed the wrapping shape once already), and (b) editing the user-facing copy in this function without remembering to update the client. The user-facing text comes from stripping the sentinel and rendering the remainder, so the function is the single source of truth for the copy.

*Client-side error mapping.* When the trigger throws, the client sees an error with `code: 'auth/internal-error'` and `message` containing the sentinel-prefixed text. The auth-screens step's `signInWithProvider` helper checks for the sentinel substring and returns `{ ok: false, kind: 'blocked-no-email', message: <stripped text> }`; the LoginForm/SignupForm OAuth handler renders `result.message` directly. No entry in `authErrorMessages` for this case.

*Emulator coverage.* `firebase emulators:start --only auth,firestore,functions` runs the blocking function in the Auth emulator. The repo's existing `npm run emulators` command should be extended to include functions so local dev exercises the trigger.

*Debounced writes.* Use an **accumulator pattern** — merge incoming partials into a ref, flush once per quiet period. A naive single-timer debounce that just forwards the latest `partial` argument would lose writes when the user types across multiple fields within the debounce window (each call would overwrite the previous `partial`, and only the last field's edit would reach Firestore). The accumulator avoids that:

```ts
const pendingRef = useRef<Partial<WellnessEntry>>({});
const timerRef = useRef<number | null>(null);

const writeWellness = useCallback((date: string, partial: Partial<WellnessEntry>) => {
  pendingRef.current = { ...pendingRef.current, ...partial };
  if (timerRef.current !== null) clearTimeout(timerRef.current);
  timerRef.current = window.setTimeout(() => {
    setWellnessEntry(date, pendingRef.current);
    pendingRef.current = {};
    timerRef.current = null;
  }, 500);
}, []);
```

On unmount or when the active date changes (the user navigates with prev/next), flush `pendingRef.current` immediately so the last keystroke isn't lost. Implementation: a `useEffect` cleanup that calls `setWellnessEntry(date, pendingRef.current)` only if `Object.keys(pendingRef.current).length > 0`, plus a parallel flush in the date-change effect. The explicit key-count predicate (rather than a truthy check) is load-bearing under React Strict Mode's mount → unmount → remount cycle in dev — without it, the first cleanup fires a no-op write of an empty object.

No external `useDebouncedCallback` dep needed — the ~15-line accumulator above lives inline in `WellnessLog.tsx` (or extracted to `src/utils/useAccumulatedDebounce.ts` if PerformanceLog wants the same shape).

*Chip state recomputation.* The chip rerenders on every input change because `WellnessLog` reads the live entry from `DataContext` (which updates via `onSnapshot` after the debounced write commits).
```ts
const { isOpen } = useNavMenu();
useEffect(() => {
  const mq = window.matchMedia('(prefers-reduced-motion: reduce)');
  let id: number | undefined;
  const start = () => {
    if (id !== undefined) return;
    if (mq.matches || isOpen) return; // don't run while reduced-motion or menu open

  };

    if (id !== undefined) { clearInterval(id); id = undefined; }
  };
  start();
  const onChange = () => { stop(); start(); };
  mq.addEventListener('change', onChange);
  return () => { stop(); mq.removeEventListener('change', onChange); };
}, [isOpen]);
```
